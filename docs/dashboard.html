01What it is

Marginalia parses a Kindle export, auto-sorts every highlight into vocab, quote, or topic of interest, and presents the library four ways. Everything lives in one page; state persists in the browser. The only server-side piece is a tiny relay that lets a phone hand a scanned highlight to the desktop.

Kindle My Clippings.txt, HTML/CSV export, manual add, or phone OCR scan

04Data model & persistence

The live globals are a projection of one active library. You never touch both copies directly — syncActiveLib() pushes globals into the library object, loadLibIntoState() projects one back out.

Every saved library. ACTIVE_LIB picks one.
⇄

saveState()
Debounced. Calls syncActiveLib() first.
localStorage
marginalia.v2 · migrates from v1
A clip = {id, batch, title, author, text, type, page, loc, added, cat, fp, edited, catLocked, review, flagged, incoming}. Its fp (fingerprint) is the stable identity used as the card id.

Rule of thumb: mutate state, then call saveState(). Never edit LIBRARIES[x].clips and STATE.clips independently.

05Import & dedup pipeline

ingest() dispatches by file type, then merges by fingerprint so re-importing the same export is idempotent — new clips add, changed text updates in place, and your own edits are protected.

06Auto-categorization

autoCategorize(c) looks at each clip and routes it. User overrides are remembered in CAT_RULES (keyed by text) and catLocked stops re-import from re-sorting them.

clip text
one highlight
vocab
single-word lookup — isVocabWord / vocabTermOf
topic of interest
proper noun / named subject — isCapCore / lookupForm
quote
everything else — a memorable passage
Keys vs labels. The data keys are immutable: vocab, quotes, topic, none. What the user sees comes from CAT_LABELS — e.g. quotes always displays as the singular "quote".

11Phone → desktop handoff (scan + OCR)

Snap a paper book's line on your phone, OCR it on-device, and it appears on the desktop library — no accounts. The token in the URL is the permission; the mailbox is one-shot and expires in 5 minutes.

Desktop · handoff.js
Generates a random token, shows a QR / link, starts polling GET /api/relay/:token.
Phone · /scan + scan.js
Opens the rear camera, captures a photo of the highlight.
Phone · Tesseract.js
Lazily loads the bundled OCR engine and recognizes the text on-device.
Phone → Relay
POST /api/relay/:token stores the JSON highlight in Workers KV (TTL 300s, ≤ 8 KB).
Relay · Cloudflare
KV holds it under the token until the first read.
Desktop poll hits
GET returns the payload and deletes it (one-shot). Clip is ingested → merged → saved.
Why a server at all? The desktop is the receiver, so it needs a real origin the phone can POST to. file:// can't be that origin — hence the single tiny relay. Everything else stays local.
